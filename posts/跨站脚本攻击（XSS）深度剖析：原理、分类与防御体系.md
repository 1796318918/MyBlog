# XSS 到底是个啥？

XSS 全称叫跨站脚本攻击，听起来挺高大上，其实说白了就是：攻击者在你的网页里插入了一段恶意的 JavaScript 代码，然后别的用户访问这个网页时，这段代码就在人家浏览器里跑起来了。

为什么这么危险？因为浏览器不知道这段脚本是网站自己写的还是被别人塞进来的，它都会老老实实执行。一旦脚本跑起来，就可以偷用户的 Cookie、把网页内容改掉、甚至假冒登录框骗密码。

# 三种常见的 XSS 类型

**反射型 XSS**：攻击者把恶意脚本放在 URL 里，比如一个搜索框，你搜索的东西会直接显示在结果页上。如果网站没过滤，我发给你一个链接 `search?q=<script>alert(1)</script>`，你点开可能就弹个窗。这种一般需要诱骗你点链接，所以利用难度稍微高一点。

**存储型 XSS**：这个就猛了。攻击者把恶意脚本存到服务器上，比如留言板里发一条包含脚本的评论。以后每个看留言的人都会中招，有点像病毒传播。当年的 Samy 蠕虫就是靠这个刷爆了 MySpace。

**DOM 型 XSS**：这个比较特殊，漏洞出在前端 JavaScript 代码里。比如页面用 `document.write` 直接把 URL 里某个参数写进页面，没有过滤。攻击者构造一个地址，`#<img src=x onerror=alert(1)>`，就能触发。全程不经过服务器，所以服务端的日志都看不到。

# 攻击者能干啥？

最常见的是偷 Cookie。如果 Cookie 没设 HttpOnly，攻击者就能用 `document.cookie` 拿到你的登录凭证，然后在自己的机器上冒充你登录。还有更高级的玩法，比如用 JavaScript 发送请求执行 CSRF 攻击、记录你在页面上的按键、弹出以假乱真的登录框等。

# 那怎么防？

1. **输入验证**：用户输入的东西一定要过滤，只保留安全的字符。特别是那些尖括号、引号之类，能不要就不要。
2. **输出编码**：把数据输出到页面上时，根据输出的位置做编码。比如在 HTML 里就把 `<` 变成 `&lt;`，在 JavaScript 里就用转义符。像 React 这种框架默认会做转义，但你用 `dangerouslySetInnerHTML` 就得小心了。
3. **给 Cookie 加属性**：设上 `HttpOnly`（让 JS 读不到）、`Secure`（只能 HTTPS 传）、`SameSite`（限制跨站请求带 Cookie）。
4. **内容安全策略（CSP）**：设置一个白名单，告诉浏览器只执行哪些来源的脚本，内联脚本一律不跑。这是最有力的防御，但配置起来有点麻烦，容易把正常功能搞挂。

XSS 本质上是数据和代码混在一起了。只要我们开发的时候时刻想着“用户输入不可信”，就能避免大部分问题。
