# OWASP 是什么？

OWASP（开放式 Web 应用程序安全项目）每隔一段时间就会发布一份最严重的 Web 安全风险榜单，也就是 Top 10。这玩意儿可以看作是安全领域的必备知识，就算你不是专业搞安全的，写 Web 应用也应该心里有数。

最新版的十大风险我就不照本宣科地念了，挑几个开发最容易踩的坑聊聊。

# 访问控制别靠前端隐藏

失效的访问控制常年排第一。常见的问题就是：前端页面不显示“删除用户”按钮，但后端接口没做权限校验，攻击者直接调接口就把数据删了。还有些通过改 URL 里的 ID 访问别人的数据（`/order/123` 改成 `/order/124`）。解决方法就一条：**权限校验必须在服务端做**，前端只是用户体验，不参与安全决策。

# 加密不是可有可无

密码存明文、用 MD5 不加盐、关键接口走 HTTP 而不是 HTTPS，这些现在看都是低级错误了。存密码就用 bcrypt、argon2 这样的强哈希算法。传输统一上 TLS，不要让数据在网络上裸奔。密钥管理也别嫌麻烦，放到专用的密钥管理服务里。

# 注入攻击依然普遍

SQL 注入我们前面单开一篇详细聊过，这里再强调一次：**用参数化查询**。别手拼 SQL 字符串，别偷懒。除了 SQL，还有 LDAP 注入、OS 命令注入，根源都是数据和指令混淆了，防御思路都一样：分开处理。

# 安全配置不能马虎

买来的服务器、中间件，默认配置通常是为方便而非安全。比如数据库默认账号 root 密码为空，管理后台 `/admin` 没有限制访问 IP。项目部署后，记得改默认密码、关掉不必要的端口和服务、去掉报错时泄露的详细堆栈信息。

# 组件漏洞别无视

现代开发大量依赖开源组件，一个应用可能有一百多个依赖。只要其中一个组件存在已知漏洞，你的应用就危险了。定期用 `npm audit`、OWASP Dependency-Check 这类工具扫描依赖，及时升级修复。别等到漏洞被利用才补。

# 日志和监控别等出事才想

很多公司被入侵几个月后才发现，因为根本没开日志或者没有人在看日志。系统上线就应该规划好日志记录什么、存多久、怎样监控异常。做到出事了能追溯，可疑行为能告警。

# 把安全想在前头

安全不应该在测试阶段才介入，那太晚了。需求和设计阶段就要做威胁建模（比如用 STRIDE 方法），开发阶段用静态分析工具扫代码，测试阶段结合自动化安全测试和人工渗透，上线后持续监控。这就是 DevSecOps 的思路，安全左移，全员负责。

背下 Top 10 的条目很容易，但真正理解每种风险怎么在自己的代码里避免，才是一个靠谱开发者的素养。
