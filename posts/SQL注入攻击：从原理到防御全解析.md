# 什么是 SQL 注入？

简单讲，就是你的程序在拼接 SQL 语句的时候，把用户输入直接当成了 SQL 代码的一部分，导致攻击者可以修改原本的查询逻辑。结果可能是数据泄露、数据被篡改，甚至整个表都被删掉。

经典的例子就是登录验证：`SELECT * FROM users WHERE name='$name' AND password='$pass'`。如果我在用户名框里输入 `admin' -- `，那么语句就变成了 `SELECT * FROM users WHERE name='admin' --' AND password='...'`。后面密码那部分直接被注释掉了，我就以管理员身份登进去了。

# 各种注入姿势

根据参数的类型和注入的位置，大概分几类：

- **数字型**：参数没加引号，比如 `id=1 OR 1=1`，直接改变逻辑。
- **字符型**：就是前面例子那种，要先闭合引号再注入。
- **搜索型**：用在 LIKE 语句里，通过 `%` 等通配符构造。
- **盲注**：页面不显示数据库错误，但可以通过页面响应的差异（布尔盲注）或者响应时间（时间盲注）来猜数据。比如 `id=1 AND IF(1=1, sleep(5), 0)`，如果页面真的等了 5 秒，说明注入成功了。
- **堆叠注入**：支持一次执行多条 SQL，比如 `; DROP TABLE users;--`，直接搞破坏。

# 手工注入是个什么流程？

假设你发现一个网站有注入点，一般会这么试：

1. 加个单引号看看页面报不报错，报错了就八成有戏。
2. 用 `order by` 猜字段个数，一直加数字直到报错。
3. 用 `union select` 看哪个位置的数据会显示在页面上。
4. 利用 `information_schema` 库查表名、列名，最后把数据拖出来。

当然，现在大家更多是用工具，比如 sqlmap，自动检测、自动利用，一条龙服务。但了解手工过程能帮你理解工具的原理。

# 防御其实很简单

**参数化查询**是根本解。不管是 Java 的 PreparedStatement、PHP 的 PDO 还是 Python 的 `cursor.execute("SELECT ...", (param,))`，都是把 SQL 结构和数据分开传，数据库不会把参数当成代码执行。

如果实在不能参数化（比如动态表名），那就要用**白名单**，只允许提前定义好的那几个值，别的统统拒绝。

还有几点辅助：数据库账号权限最小化，不要给应用 root 权限；错误信息不要爆给用户看；上 WAF 作为外层过滤。

SQL 注入最根本的原因就是数据和代码混为一谈，只要记住永远别直接拼字符串，你就能睡个安稳觉。
